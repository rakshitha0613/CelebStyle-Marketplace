**Audit date:** 2026-07-15
**Method:** Live testing against running servers (backend `:4000`, frontend `:3000`) via real HTTP requests, combined with source tracing to verify every button/handler is wired to real logic (not inert). ## ⚠️ Incident disclosure (read first)

During this audit, five parallel test passes independently found the backend crash mid-session (none of them stopped/killed anything — confirmed by process inspection). Root cause: **three duplicate `npm run dev:backend` chains and two duplicate orphaned `test:ops` processes were all running simultaneously**, left over from earlier `timeout`-wrapped commands in this session whose wrapper gave up waiting without actually killing the underlying process tree. All of them held the same embedded PGlite data directory open at once, which corrupted it — the same failure class as an earlier incident this session (`RuntimeError: Aborted()` on WAL replay).

**Recovery performed:** identified and killed all 23 duplicate/orphaned Node processes (3 backend chains, 2 test chains, 1 stale frontend instance with 9+ hours uptime and 935MB/5220 CPU-seconds), preserved the corrupted directory (renamed, not deleted — `.pglite-data.crashed-20260715154429-agentaudit`), and let the app's automatic `runSeed()` repopulate the catalogue on a fresh schema, exactly as in the earlier incident (which you already approved this recovery pattern for). **Verified restored: 101 celebrities, 100 outfits, 6 manufacturers, 4 storefronts.** All test data created during this audit (a test outfit, a test community post, several test users) was non-seed data and was wiped by the crash+reseed cycle — confirmed gone, no manual cleanup was needed. Both servers are now running as single, clean instances.

**Practical guardrail for future sessions:** never assume a `timeout`-wrapped or backgrounded `npm run dev:backend` / `npm run test:*` command has actually terminated its process tree just because the wrapper returned — verify with `tasklist`/process inspection before starting another instance.

## 🔴 Separate, more severe finding: the backend crashed a SECOND time during this audit, from a real application bug

After the recovery above, the backend crashed again on its own — this time a clean process exit (data was intact on restart), caused by an **unhandled promise rejection crashing the entire Node process** whenever a request carries an expired or invalid Bearer token to a Community or Reviews endpoint. Root cause confirmed in the backend log (`JWTExpired: "exp" claim timestamp check failed`) and in source:

- `apps/backend/src/routes/community.ts:78` and `apps/backend/src/routes/reviews.ts:77` both have the identical bug in their `optionalUserId()` helper: `const payload = verifyAccessToken(auth.slice(7));` — **missing `await`** on this async function. The surrounding `try/catch` only catches synchronous throws from *calling* the function, not the eventual rejection of the un-awaited promise it returns. When the token is expired/invalid, `verifyAccessToken` rejects asynchronously, outside the try block's protection, producing an **unhandled promise rejection that terminates the whole process** (default Node.js behavior).
- Contrast with the correct pattern used everywhere else in the codebase: both `apps/backend/src/auth/auth.middleware.ts:40` (`optionalAuth`) and `apps/backend/src/auth/middleware/authenticate.ts:30` (`authenticate`) correctly `await verifyAccessToken(...)` inside their try blocks and return a clean 401 — this bug is isolated to these two files' local reimplementation of the same "optional auth" concept.
- **Severity: critical.** Access tokens expire after 15 minutes by design (confirmed via the JWT payload's `iat`/`exp` values, 900s apart). Any real user's session naturally produces expired-token requests during normal use, and any external client can trigger this deliberately with zero authentication — it's an unauthenticated, trivially-reproducible full-outage bug on two public route groups (`/api/community/*`, `/api/reviews/*`).
- This was **not fixed** during this audit per your explicit instruction to stop after auditing — flagged here as the top item for the critical-failures triage.
